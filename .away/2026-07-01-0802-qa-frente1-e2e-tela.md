## Decisões

### D1 · 08:57 — Root cause do bug mudo: toolChoice forçado sem `prepareStep` reverte
- **Contexto:** investigação do card cross-frente. `src/lib/agent/orchestrator/detect-name-turn.ts`
  detecta heuristicamente "usuário respondeu com nome" e `runner.ts`/`index.ts` forçam
  `toolChoice: { type: 'tool', toolName: 'save_contact_name' }` no `ToolLoopAgent` (fix
  BUG-SHORT-GREETING-AFTER-NAME, intencional — força o modelo a persistir o nome mesmo se ignorar
  o prompt). Porém `builder.ts:244-259` passa esse `toolChoice` como setting ESTÁTICO do agent —
  sem `prepareStep`. A AI SDK 6 (`streamText`/`ToolLoopAgent`, confirmado lendo
  `node_modules/ai/dist/index.mjs:4332,7245`: `toolChoice: prepareStepResult?.toolChoice ?? toolChoice`)
  reaplica o MESMO `toolChoice` em TODOS os steps do loop (`stopWhen: stepCountIs(10)`), a menos que
  `prepareStep` diga o contrário por step. Resultado: Anthropic é obrigado a chamar
  `save_contact_name` em CADA um dos 10 steps — nunca pode produzir texto (tool_choice="tool"
  proíbe finish_reason=stop) — o loop esgota o teto de steps com `textChars:0`.
- **Decidi:** corrigir `buildAgent` (`src/lib/agent/agents/builder.ts`) pra adicionar
  `prepareStep` que devolve o `toolChoice` forçado APENAS no `stepNumber === 0` (preserva a
  intenção original do fix BUG-SHORT-GREETING-AFTER-NAME: obrigar a 1ª chamada) e reverte pra
  `{ type: 'auto' }` em `stepNumber > 0` (deixa o modelo responder com texto no step seguinte, como
  o comportamento saudável documentado no card: "o nome FOI salvo; o turno seguinte recuperou").
- **Alternativas descartadas:** (a) baixar `stepCountIs` — não resolve, só muda quantos steps mudos
  acontecem antes de desistir; (b) não forçar toolChoice nunca — reabre o bug original
  BUG-SHORT-GREETING-AFTER-NAME (Claude ignorava o prompt em variantes curtas); (c) forçar só via
  prompt sem `toolChoice` — já provado insuficiente (comentário em `detect-name-turn.ts`).
- **Reversibilidade:** fácil (`git revert`, mudança local a 1 função).
- **Evidência:** commit `ccbd5e7` (`test+fix:`). TDD: teste behavioral
  (`src/lib/agent/agents/builder.force-toolchoice-loop.test.ts`, introspecciona
  `(agent as any).settings.prepareStep` do agent REAL construído por `buildAgent`) +
  cassette end-to-end (`tests/regression/agent-trajectory.test.ts`, describe
  `BUG-MUTE-LOOP-NAME-CAPTURE`) — vistos FALHAR primeiro (4 asserts vermelhos, incluindo
  `settings.prepareStep is not a function`), depois corrigidos e vistos PASSAR (7/7).
  `pnpm test:unit` 219 arquivos/2204 testes verdes; `pnpm test:integration` 46/48 (2 skip)
  verdes — zero regressão.

### D2 · 09:10 — `ToolChoice<T>` de alto nível usa STRING `'auto'`, não `{type:'auto'}`
- **Contexto:** 1ª versão do fix usava `prepareStep` retornando
  `{ toolChoice: { type: 'auto' } }`. O cassette end-to-end (streamText real, não só
  introspecção) pegou que isso reproduzia O MESMO bug: a conversão interna da AI SDK
  (`prepareToolsAndToolChoice` em `node_modules/ai/dist/index.mjs`) só trata `toolChoice`
  como STRING (`'auto'|'none'|'required'`) ou objeto `{type:'tool',...}` — um objeto
  `{type:'auto'}` cai no branch `else` e vira `{type:'tool', toolName: undefined}`,
  mantendo o modelo forçado (mudo) mesmo com o "fix".
- **Decidi:** corrigir pra `stepNumber > 0 ? ('auto' as const) : opts.toolChoice`. Prova:
  script de debug isolado (`node debug-cassette.mjs` dentro do container) mostrou o
  `doStreamCalls[i].toolChoice` recebido pelo mock model em cada step — só ficou correto
  (`{type:'auto'}` de fato, texto produzido no step 2) depois da correção.
- **Alternativas:** nenhuma — é o único shape correto pro tipo `ToolChoice<TOOLS>` da AI SDK 6.
- **Reversibilidade:** fácil (parte do mesmo commit `ccbd5e7`).
- **Evidência:** `tests/regression/agent-trajectory.test.ts` describe `BUG-MUTE-LOOP-NAME-CAPTURE`,
  teste "COM prepareStep (fix builder.ts)".
- **Nota:** prova de que o cassette end-to-end (streamText real) é indispensável — o teste
  puramente estrutural/introspectivo (Camada 1) validou o CONTRATO do meu próprio código
  (`toEqual({type:'auto'})`) sem pegar que esse contrato produzia o efeito ERRADO na SDK
  real. Só a Camada 2 (execução real do loop) expôs a discrepância.
