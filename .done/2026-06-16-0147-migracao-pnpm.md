## O que mudou

A plataforma deixa de usar npm e passa a `pnpm` como gestor único, alinhando ao
padrão global TwoBrains que evita "fritar o Mac" (npm instala `node_modules`
cheio por imagem/worktree → custo linear de tempo e disco com N agentes paralelos).
Com pnpm + store content-addressable compartilhado, cada pacote existe 1× em disco
e o resto é hardlink → subir workspace novo é quase instantâneo, ~zero disco extra.

### A — Package manager → pnpm
- `pnpm import` do `package-lock.json` → `pnpm-lock.yaml`; `package-lock.json` removido.
- `package.json`: `packageManager: pnpm@11.7.0`, `engines.pnpm >= 11`, script `typecheck`
  novo, e scripts internos `npm run` → `pnpm`.
- `pnpm-workspace.yaml`: `allowBuilds` liberando os build scripts nativos

## Decisões tomadas

- **`node-linker=hoisted` (em vez de `enableGlobalVirtualStore`/layout isolado):** dois
  bloqueios reais do layout padrão do pnpm neste projeto — (1) o postinstall do
  `esbuild` extrai um binário nativo de ~10 MB que **some** quando o store fica num
  BuildKit cache mount; (2) Next 16 (Turbopack) **não resolve `next/package.json`**
  a partir de `src/app` no node_modules symlinkado; e `output: standalone` faz
  tracing de node_modules. Flat resolve os três e é o ambiente sob o qual o projeto
  foi construído. O anti-fritar é preservado: o node_modules flat **hardlinka do
  store compartilhado** → 1 cópia física global. Por isso `enableGlobalVirtualStore`
  (exclusivo do layout isolado) **não** é usado aqui — desvio documentado no
  `pnpm-workspace.yaml` e no `.npmrc`.
- **Build prod sem BuildKit cache mount no store:** o binário auto-extraído do esbuild
  não sobrevive ao mount destacando ao fim do RUN. Store no próprio layer → build
  reprodutível. Rebuild rápido continua coberto pelo cache de layer do Docker
  (install antes do `COPY . .`). O par `pnpm fetch` + `install --offline` do template
  tb-local-dev **também** falhou (não materializa optional deps nativos por plataforma
  a partir de lockfile importado de npm) — descartado pelo mesmo motivo.
- **Comando dev do projeto:** `pnpm dev` (Next dev/Turbopack), porta 3000, hostname 0.0.0.0.
- **Single-package** (não monorepo): `pnpm-workspace.yaml` existe só pra fixar `allowBuilds`.
- **Docs históricos** (`.away/`, `docs/specs/*qa-report*`, `docs/correcoes/done/`,
  `docs/test-plan*`) **não** foram reescritos — são registro point-in-time; trocar
  comandos npm→pnpm neles é revisionismo sem valor. Só docs operacionais ativos
  (README, CLAUDE.md, local-dev) foram atualizados.
