## FIX-349 — o consentimento do reveal vazava no WhatsApp

### Root cause PROVADO — DOIS bugs distintos, não um

**Bug A** (`imovel-whatsapp` t6, `servicos-whatsapp` t6): o fluxo obrigatório do reveal chama
`search_groups` e manda o modelo ANUNCIAR o resultado ANTES de chamar `recommend_groups` (que é
quem preenche `rank`, único campo que `pickBestRankedGroup` aceita pra saber "qual é o top-1").
`search_groups` já devolve administradora/parcela por grupo — se o modelo narra a "melhor opção"
nessa janela, `ctx.pendingTopOffer` ainda é `null` e `isPrematureTopOfferClaim` (o guard do
FIX-333) nem olha o segmento. Bug irmão no mesmo guard: o regex `\bADMINISTRADORA\b` nunca fechava
em nomes acentuados (ITAÚ, ÂNCORA) — `\b` do JS só entende `[A-Za-z0-9_]`.

**Bug B** (`servicos-whatsapp`, reco-consent NUNCA aparece na conversa inteira): reproduzido
byte-a-byte com o dossiê real. O modelo fecha o turno de "primeira vez" com uma pergunta genérica
sem relação com o consentimento ("Bora ver essas três opções que achei pra você?").
`EphemeralTextFilter.hasHeldQuestion()` é uma heurística CEGA — só checa se a ÚLTIMA sentença do
modelo terminou em ALGUMA pergunta, nunca se ela é sobre o gate corrente — e marca
`modelAskedGateQuestion=true`. `reco-consent` é TEXT-ONLY no WhatsApp (sem interactive) e o adapter
aplicava `modelAsked ? null : gateQuestion(...)`: sem NENHUM fallback estrutural, isso apaga o
gate por completo (nem card, nem texto — só um `console.error` mudo). O usuário nunca via a
pergunta; o consentimento acabava resolvido por um clique/menção posterior, nunca por resposta a
uma pergunta que ele viu.

### Correção
- `sanitizer.ts`: `pendingOffers` (TODAS as ofertas já indexadas, via `search_groups` OU
  `recommend_groups`, não só a de maior rank) + boundary de nome via lookaround Unicode-aware.
- `runner.ts`: popula `pendingOffers` a partir de `revealGroupsById`.
- `whatsapp/adapter.ts`: `WHATSAPP_GATES_WITHOUT_FALLBACK` (hoje só `reco-consent`) — pra gates sem
  interactive, `modelAsked` nunca apaga a única forma de entrega.

### Testes
- `sanitizer.test.ts` (describe "FIX-349", 4 casos) + `adapter.fix-349-reco-consent-silencioso.test.ts`
  (2 casos, RED→GREEN com mock de `runTurn`).

## FIX-350 — o último resquício do fallback + evasão inconsistente

### a) Fallback enlatado — root cause PROVADO por cruzamento de código

`tool-policy.ts` (`allowedTools`) nunca inclui `present_decision_prompt`/`present_topic_picker`/
`present_whatsapp_optin` em NENHUMA fase — as três viraram emissão 100% server-side havia rodadas
(FIX-253/309/280). Mas `system-prompt.ts` nunca foi atualizado: ainda instruía, com "REGRA DURA",
que o modelo CHAMASSE as três (inclusive um exemplo "GOOD" mostrando a chamada). Se o modelo
seguisse essa instrução, batia em `NoSuchToolError` → o runner suprimia toda a fala do turno →
`buildToolErrorRecoveryFallback` disparava. Precedente: o FIX-343 (bloco-e) já tinha corrigido a
MESMA classe de bug nos sub-turnos SERVER-DRIVEN via `forceToolChoice:"none"` — o resíduo desta
rodada é um turno de USUÁRIO genuíno, fora daquele escopo, causado pelo texto do prompt.

**Correção:** limpa as 3 instruções de `system-prompt.ts` (reescritas pra descrever que o SISTEMA
dispara os cards sozinho); lista de "tools idempotentes" cai de 6 pra 4.

### b) Evasão inconsistente à administradora inexistente

Guard já impedia a mentira; ninguém dava ao modelo o FATO pra responder bem. `sanitizer.ts` ganha
`findUnavailableAdministradoraMention` (reusa a lista fechada + normalização de acento que
`isHallucinatedAdministradoraClaim` já tinha); `system-context.ts` ganha `unavailableAdministradoraFacts`
(mesmo padrão de `exactnessFacts`); `index.ts` computa o fato antes do turno do modelo, só quando o
texto NÃO resolveu pra uma oferta real já exibida.

### Testes
- `system-prompt.fix-350-tools-server-side.test.ts` (6 casos) + `system-prompt.behavior-guards.test.ts`
  atualizado (as 2 REGRAS DURAs que citavam as tools removidas agora testam sem amarrar a elas).
- `sanitizer.test.ts` (describe "FIX-350(b)", 4 casos) + `system-context.fix-350b-administradora-inexistente.test.ts`
  (5 casos).
