## Achados / correções desta rodada (faixa FIX-172..176)

- **FIX-172** (bug real, INLINE): `/api/admin/simulator/attendant/[id]/reply` chamava `handleAgentMessage` (chat de vendas) DIRETO, pulando a precedência mesa-primeiro do `processor.ts` — atendente de mesa testado no simulador NUNCA falava com o copiloto (sempre caía no fluxo de vendas). Comentário da rota mentia ("routing through the same handleAgentMessage as the webhook") — o webhook real NÃO faz isso. Troca pra `processTextMessage` (mesmo entrypoint do webhook). Regressão: `route.mesa-precedence.integration.test.ts`.
- **FIX-173** (bug real, INLINE): outbound da mesa (copiloto/claim/broadcast) chamava `sendTextMessage`/`sendReplyButtons` direto de `@/lib/whatsapp/api`, sem NUNCA espelhar pro `simulator-bus` — diferente do chat de vendas (`proxy.ts`'s `sendToAttendant`, que sempre espelha). O painel dev `/admin/simulator/attendant` ficava mudo pro fluxo da mesa — bloqueava qualquer E2E de tela real do copiloto/claim. `notifyMesaAttendant(Buttons)` fecha o gap: sempre publica no bus; só pula a chamada real à Meta quando o telefone é sintético (convenção `SIM-`). Regressão: `mesa/notify.test.ts` + `simulator-bus.test.ts` (payload `interactive`).
- **FIX-174** (gap de test-infra, INLINE): não existia rota pra simular o CLIQUE no botão interativo "Vou atender" — só texto livre. Nova rota `/api/admin/simulator/attendant/[id]/interactive-reply` espelha `/simulator/whatsapp/.../send` (kind=interactive), chamando `processInteractiveReply` — mesmo claim atômico do webhook real. UI (`simulator-chat.tsx`) agora renderiza `WhatsAppInteractive` (reuso do componente do simulador de cliente) e envia o clique pra rota nova. Regressão: `interactive-reply/route.test.ts`.
- **FIX-175** (bug real de PRODUÇÃO, achado escrevendo a spec da corrida, INLINE): `getMesaAttendantList()` cacheia 60s; NENHUMA rota do CRUD de atendentes de mesa (POST/PATCH/DELETE) chamava `invalidateMesaAttendantCache()` — a função existia, nunca era invocada. Efeito real: atendente DESATIVADO continua elegível pro broadcast (recebe dossiê com dado do cliente) por até 60s depois de sair da mesa; atendente recém-criado só passa a receber depois do TTL expirar. Fix: as 3 rotas agora invalidam o cache na mutação. Regressão: `mesa-attendants/route.integration.test.ts` (novo caso "desativar via PATCH invalida o cache imediatamente").
- **FIX-176** (bug real, cosmético, INLINE): `ContactDetailPanel` e `LeadDetailPanel` tinham `STAGE_LABELS` sem `em_atendimento` (a raia nova do FIX-126) — qualquer lead nessa fase mostrava o enum cru "em_atendimento" no badge, em vez de "Em Atendimento". `kanban-column.tsx` já tinha o label certo; os 2 painéis de detalhe ficaram pra trás. Regressão em ambos os componentes.
- **Achado tangencial, NÃO CORRIGIDO — reportado (fora do escopo mesa, arquitetural)**: `POST /api/admin/attendants` chama `auth.api.signUpEmail` (better-auth) pra criar a conta do novo atendente; o plugin `nextCookies()` **sequestra a sessão de quem chama** — o Set-Cookie da sessão do usuário RECÉM-CRIADO substitui o cookie do ADMIN que disparou a criação. Um admin que cria 2 atendentes em sequência via UI real (`/admin/attendants`) fica LOGADO COMO O 2º ATENDENTE CRIADO (role=attendant, isActive=false) depois da 2ª criação, sem aviso — precisa relogar. Contornei nos meus testes usando uma sessão de setup descartável com relogin entre cada criação. Fix correto provável: adotar o plugin `admin()` do better-auth (`auth.api.createUser`, "no session is created by default") — troca de arquitetura de auth, fora do escopo desta rodada (mesa/Frente 3). **PENDENTE-KAIRO** decidir se vale a pena.
